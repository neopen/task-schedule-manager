架构设计

NeoTask 是一个嵌入式通用任务池组件，以库的形式被其他应用引用，通过函数调用方式使用。

架构概览

NeoTask 采用分层架构设计，模块化清晰，易于扩展。

分层架构详解

API Layer

提供统一的外观接口，简化使用复杂度。

  • TaskPool - 即时任务入口，专注于立即执行的任务
  • TaskScheduler - 定时任务入口，专注于延时/周期任务
  • TaskEngine - 统一引擎，协调各模块

Core Layer

核心业务逻辑层，负责任务生命周期管理。

  • Dispatcher - 任务分发器，接收任务并放入队列
  • LifecycleManager - 任务生命周期管理，状态转换
  • FutureManager - 异步等待管理，支持任务结果回调
  • EventBus - 事件总线，解耦各模块

Queue & Worker Layer

队列和 Worker 管理层，负责任务调度和执行。

  • PriorityQueue - 优先级队列，支持多级优先级
  • DelayedQueue - 延迟队列，基于时间轮实现
  • WorkerPool - Worker 池，并发控制
  • Supervisor - 监督者，Worker 健康管理

Storage Layer

存储抽象层，支持多种存储后端。

  • MemoryStorage - 内存存储，适合开发测试
  • SQLiteStorage - SQLite 持久化，适合单机生产
  • RedisStorage - Redis 存储，适合分布式部署

数据流

用户代码 → TaskPool.submit() → Dispatcher.dispatch() → LifecycleManager.create_task()
    → QueueScheduler.push() → PriorityQueue → WorkerPool.pop() → Worker.execute()
    → LifecycleManager.complete_task() → FutureManager.complete() → 用户获取结果

设计模式

Facade Pattern（外观模式）

TaskPool、TaskScheduler、TaskEngine 提供统一的外观接口，封装底层复杂组件，简化使用。

Factory Pattern（工厂模式）

StorageFactory、ExecutorFactory、LockFactory 根据配置创建相应的存储、执行器、锁实例。

Observer Pattern（观察者模式）

EventBus 实现任务事件的发布订阅，支持自定义回调，解耦各模块。

Strategy Pattern（策略模式）

多种存储策略（Memory、SQLite、Redis），可动态切换，满足不同场景需求。

Singleton Pattern（单例模式）

锁管理器、事件总线等全局组件采用单例模式，确保唯一实例。

架构演进路线图

MVP v0.1 - 基础任务池

  • 本地内存队列
  • 异步执行
  • 内存/Redis存储

V1.0 v0.2 - 监控能力 + 延时任务

  • 事件总线、指标收集
  • 延时执行、周期任务
  • Cron表达式支持

V2.0 v0.3 - 分布式核心

  • Redis共享队列
  • 分布式锁
  • 预取机制

V3.0 v1.0 - 高可用保障

  • 看门狗机制（锁续期）
  • 超时检测
  • 故障自动恢复

V4.0 v1.5 - 企业级

  • 独立Web UI
  • 多租户支持
  • Prometheus集成

设计原则

  • 渐进式增强 - 每个版本都是独立可用的完整产品
  • 向后兼容 - 新版本不破坏旧版 API
  • 可观测性优先 - 关键操作都有事件埋点
  • 嵌入友好 - 用户通过 submit()/wait_for_result() 使用
