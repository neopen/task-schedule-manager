架构设计

NeoTask 是一个嵌入式通用任务池组件，以库的形式被其他应用引用，通过函数调用方式使用。

API Layer

提供统一的外观接口，简化使用复杂度。

  • TaskPool - 即时任务入口（v0.1），专注于立即执行的任务
  • TaskScheduler - 定时任务入口（v0.3），专注于延时/周期任务
  • WorkflowEngine - 工作流编排入口（v1.5），专注于DAG任务依赖

Core Layer

核心业务逻辑层，负责任务生命周期管理。

  • Dispatcher - 任务分发器，接收任务并放入队列
  • LifecycleManager - 任务生命周期管理，状态转换
  • FutureManager - 异步等待管理，支持任务结果回调
  • EventBus - 事件总线，解耦各模块（v0.2）

Queue & Worker Layer

队列和 Worker 管理层，负责任务调度和执行。

  • PriorityQueue - 优先级队列，支持多级优先级
  • DelayedQueue - 延迟队列，基于时间轮实现（v0.3）
  • WorkerPool - Worker 池，并发控制
  • Supervisor - 监督者，Worker 健康管理

Orchestration Layer

任务编排层（v1.5），支持复杂工作流。

  • DAG Engine - DAG解析和依赖管理
  • Condition Evaluator - 条件分支表达式求值
  • ParallelExecutor - 并行执行器

Storage Layer

存储抽象层，支持多种存储后端。

  • MemoryStorage - 内存存储，适合开发测试
  • SQLiteStorage - SQLite 持久化，适合单机生产
  • RedisStorage - Redis 存储，适合分布式部署（v0.4）

数据流

即时任务数据流

sequenceDiagram
    participant User
    participant Pool as TaskPool
    participant LM as LifecycleManager
    participant QS as QueueScheduler
    participant WP as WorkerPool
    participant FM as FutureManager
    participant Storage as Storage

    User->>Pool: submit(data, priority)
    Pool->>LM: create_task()
    LM->>Storage: 保存任务(PENDING)
    LM-->>Pool: 返回 Task 对象
    Pool->>QS: push(task_id, priority)
    Pool->>FM: get(task_id)
    Pool-->>User: 返回 task_id
    
    loop Worker 调度循环
        WP->>QS: pop() 获取任务
        WP->>LM: start_task(task_id)
        WP->>Storage: 更新状态(RUNNING)
        
        alt 任务执行成功
            WP->>WP: executor.execute(data)
            WP->>LM: complete_task(result)
            WP->>FM: complete(task_id, result)
            WP->>Storage: 更新状态(SUCCESS)
        else 任务执行失败
            WP->>LM: fail_task(error)
            WP->>FM: complete(task_id, error)
            WP->>Storage: 更新状态(FAILED)
            WP->>QS: push(重试入队)
        end
    end
    
    User->>Pool: wait_for_result(task_id)
    Pool->>FM: wait()
    FM-->>Pool: 返回结果
    Pool-->>User: 任务结果

工作流编排数据流

sequenceDiagram
    participant User
    participant WF as WorkflowEngine
    participant DAG as DAGEngine
    participant Pool as TaskPool
    participant Storage as Storage

    User->>WF: submit_workflow(definition)
    WF->>DAG: build(nodes)
    DAG-->>WF: 验证通过
    WF->>Storage: 保存工作流定义
    WF-->>User: execution_id
    
    loop DAG 执行
        WF->>DAG: get_ready_nodes(completed)
        DAG-->>WF: ready_nodes
        
        par 并行执行
            WF->>Pool: submit(node.task)
            Pool-->>WF: task_id
            WF->>Storage: 更新节点状态
        end
        
        Pool->>WF: on_node_complete(node_id, result)
        WF->>WF: evaluate_condition(node)
        
        alt 条件满足
            WF->>WF: store_node_result()
            WF->>DAG: mark_dependency_satisfied()
        else 条件不满足
            WF->>WF: skip_downstream_nodes()
        end
    end
    
    WF->>Storage: 更新工作流状态
    WF-->>User: 工作流完成

分布式锁时序图

sequenceDiagram
    participant Node1 as Node 1
    participant Node2 as Node 2
    participant Redis as Redis
    participant WD as WatchDog

    Node1->>Redis: SET lock:task:123 NX EX 30
    Redis-->>Node1: OK (acquired)
    
    Node1->>WD: 启动看门狗
    
    loop 每 10 秒
        WD->>Redis: GET lock:task:123
        Redis-->>WD: Node1
        WD->>Redis: EXPIRE lock:task:123 30
        Redis-->>WD: 1 (续期成功)
    end
    
    Node2->>Redis: SET lock:task:123 NX EX 30
    Redis-->>Node2: NULL (获取失败)
    Node2->>Node2: 等待或放弃
    
    Node1->>Node1: 任务执行完成
    Node1->>Redis: DEL lock:task:123
    Redis-->>Node1: 1 (释放成功)
    
    Node1->>WD: 停止看门狗

设计模式

Facade Pattern（外观模式）

TaskPool、TaskScheduler、WorkflowEngine 提供统一的外观接口，封装底层复杂组件，简化使用。

Factory Pattern（工厂模式）

StorageFactory、ExecutorFactory、LockFactory 根据配置创建相应的存储、执行器、锁实例。

Observer Pattern（观察者模式）

EventBus 实现任务事件的发布订阅，支持自定义回调，解耦各模块（v0.2）。

Strategy Pattern（策略模式）

多种存储策略（Memory、SQLite、Redis），可动态切换，满足不同场景需求。

Singleton Pattern（单例模式）

锁管理器、事件总线等全局组件采用单例模式，确保唯一实例。

Builder Pattern（建造者模式）

WorkflowDefinition 支持链式构建 DAG 工作流。

设计原则

  • 渐进式增强 - 每个版本都是独立可用的完整产品
  • 向后兼容 - 新版本不破坏旧版 API，存储格式支持平滑升级
  • 可观测性优先 - 关键操作都有事件埋点，指标默认收集
  • 嵌入友好 - 用户通过 submit()/wait_for_result() 使用，无需理解分布式细节
